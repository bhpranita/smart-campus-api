# Smart Campus Sensor & Room Management API

## Overview

This RESTful API was created for use in the Smart Campus Project of our university through the use of JAX-RS (Jersey). The facility management team will control the rooms and sensors around the campus via RESTful API. The system's data is kept by HashMap and ArrayList running under the Grizzly HTTP server.

## What I Have Done

- Set up a Maven project with Jersey and Grizzly embedded server
- Configured the application entry point with @ApplicationPath("/api/v1")
- Implemented a Discovery endpoint at GET /api/v1 returning API metadata
- Built full Room management (GET all, GET one, POST, DELETE with safety check)
- Built Sensor management (GET all with optional type filter, POST with roomId validation)
- Implemented Sub-Resource pattern for Sensor Readings
- Created custom exceptions and mappers for all error scenarios
- Implemented a logging filter for all requests and responses
- Used DataStore class with ConcurrentHashMaps for all in-memory storage

## Conceptual Report

### Part 1.1 — JAX-RS Resource Lifecycle

Each time there is a new request, an entirely new object of the resource class is generated by the framework. This is known as the per-request lifecycle. This indicates that no data can be stored within the resource class because it is instantiated afresh for each request.

This was extremely significant for our application since we were using memory instead of database storage for data. In case I continued to use room and sensors as the instance variables of the resource class, the data would be cleared out for every request. Thus, I created a separate DataStore class and stored the data using static ConcurrentHashMaps within the class.

The thread-safety aspect could also be considered here. Several requests may be received concurrently and will operate on the very same instance of the DataStore class; this means that they are exposed to race condition problems. That is the reason why I decided to use ConcurrentHashMap in lieu of the simple HashMap implementation.

### Part 1.2 — HATEOAS

HATEOAS stands for "Hypermedia as the Engine of Application State." This means that when there is a response from the API, it is supposed to come with links that show the next steps that one can follow. This means that after getting details on a specific room through the API response, one should also be able to delete the room and access its sensors.

This is seen as advanced REST architecture for the reason that this API becomes self-documenting. The client no longer needs any external documentation to know which resources exist and how to utilize them as everything that the client needs is contained within the API's response. In addition to this, if at any time the server decides to change its URL structure, it would not affect the client because the client uses the links within the response rather than absolute URLs.

### Part 2.1 — Returning IDs vs Full Room Objects

If you only return IDs in a list, the response is very small and fast to transfer. But the client then has to make a separate GET request for each individual room to get the actual details. So for 100 rooms that means 100 extra requests which is really inefficient.

If you return the full room objects instead, the initial response is bigger but the client gets everything they need in one go without making any extra calls. For our Smart Campus API I went with full objects because facilities managers need to see all room details at once. Forcing them to make 100 extra calls just to see room names would be a terrible experience.

### Part 2.2 — Is DELETE Idempotent

YES Delete operation is idempotent in my project. Idempotency refers to performing an operation repeatedly without getting different results. When calling for the first time DELETE /api/v1/rooms/LIB-301, the room will be deleted and response code 200 OK will be returned. The subsequent calls will return response code 404 Not Found because room no longer exists.

Despite the response status being different from the first call to the second, the state of the server remains unchanged in both scenarios, as the room no longer exists regardless of the call made. This is precisely what idempotent refers to in REST, and it implies that DELETE requests can be reissued safely without doing harm.

### Part 3.1 — Wrong Content-Type Consequences

By using @Consumes(MediaType.APPLICATION_JSON), we make JAX-RS understand that this endpoint will only accept JSON data. Therefore, when a client makes a request with text/plain or application/xml data, JAX-RS discards this request without executing any line of code from our side by sending a 415 Error.

This is very helpful since we no longer need to manually check the format of the incoming request using the code itself, but let the framework handle that for us. The error message that the client receives clearly tells them about the issue, allowing them to figure out how to correct it. In the absence of this annotation, our JSON parser will raise an exception, leading to a 500 error response rather than a more useful 415 error.

### Part 3.2 — @QueryParam vs Path-based Filtering

Using @QueryParam for filtering like GET /api/v1/sensors?type=CO2 is much better than putting the filter in the path like GET /api/v1/sensors/type/CO2 for a few reasons.

First of all, the parameters are optional anyway. So, if you do not specify type, you simply receive all sensors, which is what you need. In the case of type being included in the path, there would be another endpoint required for retrieving all sensors regardless of type.

Secondly, query parameters are the usual method for applying filters within REST APIs. Everybody who has ever worked with an API knows that query parameters imply filters. Placing the filter within the URL path causes type/CO2 to appear as a separate resource, which is misleading since it is merely a filter criterion.

Third, when you want to include additional filters like "status" or "roomId" later on, using query parameters simply involves concatenating those parameters in the format "?type=CO2&status=ACTIVE". Path based filters make it very hard to do that.

### Part 4.1 — Sub-Resource Locator Pattern

This pattern suggests that instead of processing all routes within a single class, nested routes will be handled by a different class. For example, in our application, we have SensorResource for processing /sensors, but for any incoming route like /sensors/{sensorId}/readings, we process it using the SensorReadingResource class.

The major strength is that the code will remain readable and organized. If everything were to go into one file, it would take several thousand lines of code, which would be impossible to organize. With splitting based on resources, every class would have its own role.

This makes it easier to perform tests as you can test RoomResource, SensorResource, and SensorReadingResource in isolation without affecting one another. When working with real-world API having many endpoints, using this technique is almost a necessity due to sheer complexity involved.

### Part 5.2 — Why 422 Instead of 404

If the client creates a sensor through a POST call with a non-existing roomId, then using the status code 404 Not Found will be inappropriate because 404 implies that the URL itself could not be found, while in the current situation, there is nothing wrong with the URL.

The HTTP 422 unprocessable entity is far more precise since it informs the client that the request was processed by the server without any issues, the syntax was correct, but there is a problem with semantics. The specific issue is that the roomId passed in is not valid within our application. This immediately lets the client know that they should address the issue with their input data.

### Part 5.4 — Cybersecurity Risks of Exposing Stack Traces

Showing stack traces from Java to any clients of your API is extremely bad from a security perspective, because there's actually a lot you can learn from a stack trace about what your code does internally.

As an example, a stack trace gives you the exact name of classes, functions, and even lines in your source code. This means that an attacker can easily figure out the structure of your software and take advantage of its weak points. Furthermore, the attacker can get an idea of which third-party libraries or frameworks you are using and even their version numbers.

In addition, stack traces can sometimes hold the value of variables at the point of occurrence of an exception, which can inadvertently result in leakage of sensitive data related to users or the application. That is the reason why we always catch all the exceptions through our GenericExceptionMapper and respond with a 500 status code.

### Part 5.5 — Why Use Filters for Logging

The use of a JAX-RS filter for logging is significantly more effective than writing Logger.info() calls within each resource method.

This happens because the logging code only needs to be written once. In the case of manual logging, the logger calls need to be added every single time a new endpoint is created. This will be forgotten at some point by somebody, and you will have inconsistent logs for the entire API.

The filters get applied automatically to all requests and responses once they are configured. It is one-time setup and works anywhere you go. This is referred to as a cross-cutting concern, and JAX-RS filters are meant for such concerns.

And even if you have to alter the logging format and include additional details in your logs in the future, you would be able to do it from just one point in your program as opposed to searching through all resource classes to locate each Logger.info() method.
